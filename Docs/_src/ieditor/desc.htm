IEditor library

Insanity Editor is a library build on top of all lower-level libraries. It uses other libraries in similar way as an application does, and provides helping hand for game application during development. The editor DLL is intended to be linked to game executable only at development time, and not to be shipped on final CD.

Thanks to this approach, most editing code may be away of main game executable, not consuming resources of final product. Also, the level of testing and maintaining is lower than for final game, because the editor is usually used only by development team. Besides all this said, the editor is quire mature program, providing many useful tasks, without which the development of the game would become practically impossible.

Library: IEditor.dll, IEditor_d.dll
Header file: I3D\Editor.h
Creation function: CreateEditor

 

Editor architecture

The editor is based on a master interface C_editor, which performs bookkeeping and overall management, but little of actual work. It is used for initialization of plug-ins, dispatching of messages, managing shortcuts, menus, and similar tasks. Around this master interface is built a system of plug-in modules, which is very extensible. The interface defines base class for all plugins, which provides basic communication interface among the plugins, and way how plugins may receive information. Specialized plugins, performing their specific tasks, provide other methods for their controlling. The editor is designed a single DLL, where most built-in plug-ins reside. Anyway, plug-ins from other modules (e.g. from main game module) may be used too.

Each plugin is uniquely identified by its name. Although it seems that each plug-in is independent entity in the system, there’s certain dependence, mainly on core plug-ins. For example, most plugins require information about current selection. Thus the “Selection” plugin is essential for work of any other plugin which requires selection. It is assumed that several core plug-ins are always present. Anyway the suggested behavior of plugins is not to assume that particular plugin exists. Usually, plugins search for dependent plugins they use during their initialization, and if they fail to find essential plugins, they return fail code, telling the master interface that they were not initialized correctly, and will not be used.

The plugin system allows adding new plugins or removing unnecessary ones (but not in run-time). Game programmers may develope new plugins performing specific tasks, such as animations, scripting, etc.

 

Interfaces

There’re two main interfaces used in the system:

C_editor – is one and only for entire game. It is the editor itself, using plugins to do the work.
C_editor_item – base class of single plugin. Plugins usually specialize this class, and may provide public interface so that other plugins or parts of game may use them. Anyway, for many plugins, the interface of the base class is enough for complete communication with the rest of the system. The communication is done by installing keyboard shortcuts, menu items, and processing Tick and Render functions, by which plugin may perform animated tasks, or specially render its output.

Plugins

The plugins are identified by their name (string). Knowing name of a plugin, you may search for its specialized interface in the editor header file. To find particular plugin, use C_editor::FindPlugin method, using the name of searched plugin. Once you find desired plugin, you may downcast it to its real interface in order to use special methods it provides.

Controlling the editor

The MouseEdit plugin provides main interface for managing the scene. This paragraph discusses modes how user may navigate in the scene and perform basic modificatons.

The work in Editor is performed in two editing modes – a View mode and an Edit mode. By switching between these two modes, a designer may navigate freely through the mission and simultaneously edit objects in the scene.
