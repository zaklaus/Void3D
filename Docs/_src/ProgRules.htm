Programming conventions

 

 

This section discusses programming conventions used in Insanity3D engine.

 

 

1.      Names

 

Following naming rules are used in Insanity3D libraries:

 

1.1 Names of function

 

Function names are in lowercase, except of first letters of each word in the filename, for example:

 

void GetSum();

int ProcessNumber(float);

 

 

1.2 Names of variables

 

Names of variables are in lowercase, words are separated by underscore. The underscore is not used at beginning or end of name (because such names are C++ reserved).

Boolean variables express exactly what they mean, and should not be ambiguous.

Examples:

 

int total_amount;

float max_speed;

bool is_evaluated;

 

 

1.3 Names of constants

 

Constants are all uppercase, words are separated by underscore. The same apply to constants defined by preprocessor macro #define.

Examples:

 

const float NUM_ELEMENTS = 2;

#define TEXTURE_SHIFT_BITS 16

 

 

1.4 Names of enums

 

Names of enums are similar to constants, but contain common prefix describing the enumeration.

Example:

 

enum E_SIMULATION{

   SIM_UNKNOWN = -1,

   SIM_CREATE_ACTOR,

   SIM_EMIT_SOUND,

   SIM_LAST

};

 

 

1.5 Names of structures and classes

 

Structures and classes are declared without use of typedef keyword. Structure or class name directly follows the struct or class keyword.

Structure names are prefixed by S_, and class names are prefixed by C_. The rest of name is all in lowercase, with underscore separating words.

Example:

 

class C_bit_field{};

struct S_search_item{};

  1. Comments

 

2.1 Function declaration

 

Function is commented at point of declaration - that means in most cases in a header file. Class member functions are commented in class declaration. Static functions are commented when they are defined. Function declarations are separated by empty line and a separator line: //-----.

Example:

 

//----------------------------

// This function returns square root of input parameter 'f'.

   float Sqrt(float f) const;

 

 

2.2 Inline comments

 

Comments made inside of program are in eol-comment style (beginning with //), so that blocks of code may be easily commented out. They start on column 31, so that they are not mislead with commented-out lines of code.

Example:

 

 

                              //reserve space for len

   c.write((const char*)&nlen, sizeof(dword));

                              //allocate storage for file contents

   char *buf = new char[nlen];

 

 

2.3 Comments of variables, enums, constants

 

Variables and enums, if their description is trivial, are commented on the same style as described in point 2.2. If their description is longer, and would get out of the screen (screen is considered to be approx 100 columns wide), they are commented immediatelly before variable is declared, and the comment may split into multiple lines.

Comments are never put behind the lines of code they're commenting.

Examples:

 

class C_fool{

   int num_fools;             //total number of available fools

 

                              //boolean values describing if particluar fool is

                              //absolute (true) or only moderate (false)

   vector<bool> absolute_fools;

};

  1. Using 3rd party libraries

 

3.1 Using STL (standard C++  library)

 

Insanity libraries heavily uses C++ standard libraries. The code is compatible with Microsoft Visual C++ 6 and .NET implementations.

 

 

3.2 Using other 3rd party libraries

 

For some algorithms, Insanity3D uses third-party freely available libraries. Usually they are used inside of modules, not exposing this fact to user of the library. So user doesn’t need to care about installing any 3rd party library in order to compile Insanity3D source code.

 

 

 

  1. Library architecture

 

4.1 Library interface

 

Insanity uses style of hidden implementation, with exposing only well-defined public interface. There’re several modules, which perform very specialized tasks (such as script engine, sound engine, etc). All communication with the modules is done through several classes defined in a header file. The header file usually contains only one function, which creates master class, and all the other classes are created using the master interface. The methods of the interfaces are accessible using virtual functions, which operate on a pointer into the class. The header files usually do not expose any data members of complex classes, so implementation of the classes in libraries may change without need to change program which uses them (of course if some method is removed or its parameters modifies, some changes in code may be required).

 

 

4.2 Source code

 

Provided game source code also uses hidden implementation of some classes. Some classes are partially hidden – they expose part of its class declaration with several data members in a header file, and providing virtual functions into more complex methods. The implementation is of such class is then done in single .cpp file. Some classes, inherited from a common base class (for example actors) may even not expose their public interface, if there’s no need of specialized communication with such classes.

Classes providing the implementation of such defined interface have usually _imp postfix. For example, if public class interface has name C_bird, the implementation of the class in a CPP file is C_bird_imp.

Note that classes hiding its implementation are typically accessible only using its pointers, and calling virtual functions on the pointer.
