Formats and importing

Exporting from modeling package

Here we discuss how exporting from 3D modeling package, such as 3D Studio MAX is used with Insanity engine to import the data.

Materials and map slots

Following tables describe material properties used from 3DS MAX in Insanity3D engine.

This table shows how specialized material behavior is encoded into material name. The Insanity loaded parses the material name, and if mentioned character is found anywhere in the name, the behavior of the material is adjusted.

Parameters of material (encoded in material’s name):

Mark

Description

!

The texture will be drawn color-keyed, with color key being the first color in palette (for palletized textures), or black color for RGB textures.

!!

The same as above, but this will cause using zero alpha-ref value, which will improve mipmapped colorkeyed textures. Suitable for fences and textures with thin opaque pixels, which would disappear in lower mipmap levels.

@

Keep original diffuse color of material (as set in material editor), also if material has texture assigned. If this flag is not set, diffuse color of material is set to white.

/<id>

Material will be used as mirror. Specified ID causes similar materials be grouped and rendered in one pass.

%%

Reflection bitmap specifies cube texture map (not spherical map, which is the default). The cube map is made of six textures, which names are determined by name from map slot, but four appendices are added before loading: “_l”, “_r”, “_b”, “_t”, “_k” and “_f” for left, right, bottom, top, back and front sides of the cubemap.

^

Do not check this material for duplication test during export – this marks an explicit duplication due to some reason not know to the exporter plugin.

=

Material will be rendered using ADD mode. This means that resulting RGB values will be added to the content of backbuffer.

 

Similarly, additional parameters are encoded into name of map in the map slot of material.

Properties of material map (encoded in map’s name):

Mark

Description

&<proc_name>

Use procedural texture for this map (no need to specify bitmap in this case).

+         

Choose true-color pixel format for texture, if available on hardware.

$-

Mipmaps will not be created on loaded bitmap.

c-

Disable the compression of this map, if compression is set as default during loading a scene.

|<speed>

Animated sequence. The bitmap name specifies last image in the sequence. The name must end with a number. A sequence is made from image of the same name, with number changed to zero up to the number specified in the name (for example, image name may be “water31.png”, which will load 32 images with name “water??.png”, where ?? is replaced by actual image index.
Speed specifies delay, in milliseconds, how long each image will be displayed.

Blend ratio of Ambient and Diffuse maps. Applicable only in some shaders.

Object properties

Each object in 3DS MAX may have assigned user-defined properties. These properties are used by the engine for special behavior applied during import of the data file.
The properties are defined by a keyword followed by optional parameters. Following table describes all supported keywords and their parameters.


Example of user-defined properties in 3DS MAX.

More keywords may be used simultaneously. Entire block must be enclosed in {} braces. Keywords and parameters are separated by space. If some string parameter contains spaces, it may be enclosed in quotes.

Keyword

Applies to type

Description

SNGM

visual

The visual will be cast to single-mesh object, and will be initialized using all joints attached to it.

PRTC

visual

The visual will be cast to particle. Only material from original geometry will be used, so it’s convenient to apply this keyword to single triangle to use the material attached to it.

LMAP

visual

The visual will be cast to light-mapped object.

BBRD

visual

The visual will be cast to billboard.

MRPH

visual

The visual will be cast to morphing object. Any regions attached to it will be used for definition of morphing regions.

FLRE

visual

The visual will be cast to flare object. Only material from original geometry will be used, so it’s convenient to apply this keyword to single triangle to use the material attached to it.

OUVS

visual

The visual will be cast to uv-shifting object. All uv coordinates of this object will be shift by parameters assigned to the visual.

SECT

visual

The visual will become part of sector, and its geometry will be used for building the sector. If the object has parent with this flag set, its geometry will be used for building the sector on top of parent. Otherwise new sector will be created, where this object will be linked.

help

sector

Help sector wall, this object will be used only for building the sector, and won’t be put into hierarchy tree. Used with SECT keyword.

simple

sector

The sector will have no portals, and will be used only as logical container of sounds and lights. Used with SECT keyword.

one_way_portal

visual

The object defines a one-way portal. This keyword works only with SECT keyword. The center of object is used to search closest portal in the sector, and the sector will be seen through only one way (outside of sector).

VOLS

dummy

The dummy will be converted to spherical collision volume.

VOLR

dummy

The dummy will be converted to rectangular collision volume.

VOLB

dummy

The dummy will be converted to box collision volume.

VOLP

dummy

The dummy will be converted to capped-cylinder collision volume.

JTBB

dummy

The dummy will be used as bounding-box of parent joint. It will not be put into scene hierarchy.
This keyword is effectively interpreted as ‘region’ keyword, and is used for compatibility. Use ‘region’ keyword instead.

JTVB

dummy, single-mesh

The dummy will be used as volume-box of parent joint or single-mesh object. This volume box is used for collision testing on single-mesh characters. The dummy object will not be put into scene hierarchy.

region [n]

dummy

The dummy will be used for specifying region. The region size is determined from bounding box of the dummy. The region usage is dependent on type of parent object. The dummy object will not be put into scene hierarchy.
Optional index value may be specified, which is used depending on type of parent frame.

smooth

visual

Normals of all visuals with this flag will be smoothed in order to assure smooth connection between adjacent objects making one complex object. This flag is mostly suitable for terrain.

hide <n>

visual

The visual won’t be rendered beyond specified distance from camera. The distance is specified as parameter n. This distance is used at camera’s FOV set at 65 degrees, for different FOV this distance is scaled appropriately.

joint

The joint’s linked vertices will be rendered using simple mode (no joint children will be applied to vertices) beyond specified distance. This keyword may be used only together with break.

material <n>

visual, volume

Assignment of logical material for visuals or collision volumes. The material ID is passed in parameter n, and specifies ID number of applied material.

auto_lod <a b c d>

visual

Using this keyword, automatic LODs (level-of-detail) are computed for specified visual object. Parameters are treated as follows:
a … minimal distance from camera, until which best quality is used
b … maximal distance from camera, beyond which worst quality is used
c … number of LOD steps to compute (reasonable number is around 8)
d … number of faces for worst LOD

procedural <name>

visual

Procedural to be attached to the visual. Only some types of visual objects accept procedural parameters (for example morphed visuals).
The name specifies name of procedural to attach to the visual.

lock

joint

Lock joint’s position, so that it cannot be changed later. This flag is suitable for joints, whose position is assumed not to change (joint which is part of a skeleton), and entire animation is performed by rotation or scale.

break

joint

The joint breaks bone hierarchy into logical segment. All geometry affected by joint children of ‘break’ joint will be rendered separately, which allows more joints be used on same single-mesh visual.

param <index value>

visual

Set visual’s parameters at load-time.
The keyword accepts 2 values:
index … index of parameter to set
value … value of parameter
The index may be either integral zero-based index of property, or string specifying the name of property.
The value may be either number (integer or float), or name of enumerated type.

An example of properties block:
{ BBRD
param “Axis mode” true
param “Rot axis” Z
}

This causes visual to be loaded as billboard, with appropriate billboard parameters being set.

File format

Insanity engine uses two main file formats:

.i3d – used for exporting of data from 3D modeling package (3DS MAX) into internal format. Exported data are fully importable using the import plugin.
.bin – binary data used to keep scene information edited by Insanity editor

Both file formats are chunk-based, accessed using the C_chunk class declared in C_chunk.h header file. Every data in the format reside in a chunk, starting with a 6-byte header, which contains following information:

struct S_ckhdr{
   word type;
   dword size;
};

type is user-defined chunk identification number
size is the size of chunk, in bytes.

Following the header are actual chunk data. It is easy to skip unknown or unnecessary chunks in a file, while still keeping format compatibility. The methods of C_chunk class provide convenient access to chunks, and reading/writing most common types directly, such as integers, floats, etc.

The chunks of.i3d file format is described in I3D\I3D_format.h header file. The chunks of the .i3d file format are defined in I3D\Insanity.h header file, along with common types used in a game – resource caching classes, and similar helper classes.
