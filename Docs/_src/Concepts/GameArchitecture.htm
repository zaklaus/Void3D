Architecture of a game
using Insanity engine (proposal)

 

Introduction

 

This is a case study about architecture of possible game using Insanity engine. The game skeleton provided in the SDK uses the architecture described here. Also, this architecture is being used in real shipping projects.

To begin, you may start examining provided SDK skeleton application.

 

Note that all information provided here is just a hint, not actual reference, how Insanity3D must be used. The engine is suitable for building a variety of game genres, utilizing various parts of the engine in many various ways, not just the way, how we describe it here.

This proposal is based on projects, on which Insanity was being grown, and which prove to work during the development of the engine. You may decide how, and what to use from provided information.

The application is initialized, and after it, it enters into its main loop, in which it stays until it doesn’t exit. Before exiting, the application performs clean-up tasks.

The main loop is a function with a very few lines of code, which just performs tasks shown above:

  • Read input (keyboard, mouse, controller).
  • Update editor. This is done only during development, when editor is linked to the game application.
  • Update active game class
  • Render screen

 

 

Active tick class

 

The arrow switch on the image above means that focus of active class is changed using a pointer, and for the main loop, the pointed-to class doesn’t matter. There’s a global variable – pointer into currently active class, which absolutely controls the flow of the game at the moment. This class inherits from C_class_to_be_ticked class, which may be defined as follows:

 

class C_class_to_be_ticked: public C_unknown{

public:

   virtual ~C_class_to_be_ticked(){}

 

//----------------------------

// Update - to be called once per frame.

   virtual void Tick(const S_tick_context&) = 0;

 

//----------------------------

// Render frame.

   virtual void Render() = 0;

};

 

The class inherits from C_unknown, which means that it is reference-counted. So if there’s active class in the system, it is kept until someone keeps reference to it.

There’re two main methods of the class: Tick and Render, which means that the classes inheriting from this will be able to update themselves and render the results of their work. The methods are declared as pure virtual, so that inherited classes must provide them.

 

There’s a single global variable in of this type the game called tick_class, which may be written to by currently active tick class. When this variable becomes NULL, it’s signal for the main loop to jump out and proceed to exit.

 

 

Types of tick classes

 

The tick classes provide any functionality in the game, which manages game control for particular time. It may be main menu, briefing, or the game mission managing actual game play. Since each of classes performs only particular work, their implementation may be hidden in single .cpp file. There may be very simple tick classes such as mission loaders, giving control to other tick class as soon as their work is finished, as well as very complex tick classes, as the one managing the game play.

 

The tick classes are created and destroyed at runtime as necessary. There doesn’t need to be any tick class allocated as global variable. As soon as the active tick class decides that it is time to give control to other tick class, it simply creates required tick class, and stores it into the global tick_class pointer. The assignment may effectively decrement its own reference counter and release it from memory right after the assignment.

GameMission tick class

 

One of the most important tick classes is the one managing game play. We’ll call it C_game_mission. If editor is loaded, this class must be created at beginning, because editing occurs using also variables in the gamemission class.

Assets managed by this tick class may basically include:

·         checkpoint grid

·         actors

·         game setting tables

·         scripts

 

For example, the gamemission class holds actors, which are present in the mission when the mission begins. So the actor container is edited directly using the gamemission class.

 

Editing of game mission contents

There’re few editor plugin incorporated directly into the game sources, which know about how C_game_mission works (through its header file), and helps in editing of such things, such as actor assignment, script assignment, checkpoint creation, table editing, etc. Since all the editing is necessary only during development, these plugins (similarly as the Editor) is not included in the final game, because the mission files are just loaded in final game and used in game play.

 

Tasks of game mission tick class

The game mission class performs mainly two basic methods defined in the C_class_to_be_ticked base class: update and rendering. Durint the update (or Tick), the class updates all its internal data which require updating, mainly:

·         actors

·         effects (weather, screen effects, etc)

·         script manager

·         etc

 

Also, the responsibility of the tick class is to keep current game state (and possibly expose it to outside using the screen output, or using script functions, so that actors or scripted frames may learn it and make decisions upon this information).

The task of the game mission class is also to decide when the game is finished – successfully or unsuccessfully (it may use script system for this purpose), and provide appropriate action – proceed to next mission, load main menu, etc.

 

 

Actors

 

An actor is a fundamental active entity in the game mission. Actors are classes inheriting from common base class named C_actor, which perform wide variety of game functionality. The actor types are devised during game development (usually by programmers), and their behavior is had-coded in the game source files. Following are examples of actor types:

  • Player (processing player’s input, moving player’s avatar)
  • Human – representing enemies and NPCs
  • Vehicle
  • Resource (weapons, ammunition, key, etc)
  • Gunshots
  • Physical objects
  • Door
  • Effects

 

This is just a brief list, anyway it shows how big variety of functionality an actor may represent. The base actor class have following properties:

  • actor type (for identification and casting)
  • name
  • frame (3D object on which it operates)
    this gives it another common properties: position, orientation, scale

Basically, these properties is all what needs to be known about generic actor. Specialized actors have their own properties and methods exposed using their specialized interface defined in a header file.

 

Also, each actor gets possibility to update its state in each main loop (this is managed by gamemission tick class), using the actor’s Tick method.

 

Actor creation, lifetime and destruction

 The lifetime of actors is also very relative. Some types of actors may be created directly in the editor (such as NPCs, vehicles, etc), some neither (gun shot, projectiles). Some actors live in the mission for entire time of played mission (such as main player, NPCs, etc), while some are destroyed as soon as their work is finished (circle on a water level). Some actors may be created in run-time (for example gun-shoot actor, created each time when player fires). The creation is performed in appropriate place, for example enemy actor may create projectiles in the moment of firing from his weapon (furthermore, the enemy actor doesn’t care about fired projectile at all, since fired projectile actor flies on its own, and is destroyed as soon as it creates explosion actor).

 

Communication among actors

Actors may communicate in a variety of ways. Knowing the vector of all actors in the mission, particular actor may search closest actor and use it for particular purpose (for example player actor may search for door actor nearby when player hits Use key, and call its Open method to open the door). Some communication may be predefined by designers – for example actors may be controlled using scripts to perform particular action on other actors, while the actions are defined in a script file.

This means that actors which are designed for communication with other actor may search them by name, by their position, type, etc.

 

 

Scripting

 

The role of scripting in the game is to give possibility for variations, using decision making encoded in scripts. Scripts are not used for programming game behavior, which should be implemented using the actor system. This is due to various reasons:

-          script code runs slower than compiled C++ code (by ratio about 1:8)

-          scripts are designers’ tool for incorporating the story into missions

-          scripts are not programmers’ tool for performing their programming tasks (they usually use professional compilers for this)

-          script system is designed to be safe (no pointers, no arrays), so that designers never make the game to crash

-          script system has far less information about game internals, than game source code provides

 

Now when we know what script is not used for, here’s what it is intended for:

 

Calling C++ functions

Script may call functions in C++ code with various parameters. The C++ compiled functions may optimally perform desired action (for example flashing object’s brightness for several seconds), as well as they may access anything inside the game. Anyway game programmers are responsible for writing and debugging the code.

Script’s header file exposes definitions of the callable functions, their default parameters, enums, etc.

The C++ callable functions should be very strict when performing checking on input parameters, and output every possible fault into log, so that designers are allowed to make errors, and more importantly knows about them.

 

Keeping global game state

The script system is capable to holding a global game state (using global master script), which contains variables keeping its values from mission to mission. This allows putting global items into the script’s memory, for example knowledge if player killed particular character in one mission, and use this information several missions later.

 

High-level decision making

Scripts are used for higher-level mission scripts. They’re usually very unique from mission to mission, and they control the game flow. There’s usually not much time spent in the script code.

 

Frozen scripts

Scripts may be frozen when performing particular action. For example there may be script function named Pause, which pauses script’s code, and resumes it when specified pause expired. There may be more important functions than this – for example function Detect, which pauses script until defined action is not detected, and then moves script on, performing particular action depending on the detection.

The scripts may be frozen only inside of called C++ functions. For example, the C++ function implementing the Pause method freezes the script, and when there’s time to move on, the script manager runs the script from the frozen position.

 

Scripts used by actor

Scripts may be utilized for performing actions on actors. Let’s illustrate it on most advanced case – enemies. The AI may be controlled using the script. Anyway, it doesn’t mean that entire AI is possible to make in the script. Rather, the scripts are tools for designers to affect AI decisions. At game begin, the script may setup AI’s basic program – process simple commands such as MoveTo, Guard, LookAt, etc, to instruct AI to perform customize behavior. With use of script tables, it allows very customized behavior, with good amount of reusing the same script file for multiple instances of AI.

This was definition of default AI behavior. To make decision blocks, programmers may decide to call some callback script functions, for example every time when AI spots an enemy. They script may perform customized behavior on such action.

Scripts may also be used for sending signals to other AI’s at appropriate time, and perform actual decisions.

 

 

 

AI

 

AI is one of most important game element. It distinguishes good games from bad games. It is also hard to make right and convincible. Since there’s much study about this subject, we’ll come through some basic ideas which may help to make interesting AI, but do not grant it will be better or worse from other games’ AI.

 

Path-finding

AI navigation is task solved by use of navigation checkpoints placed in the mission. The comfortable checkpoint plugin allows easy definition of checkpoints and their connections. The result is a grid of connections, which define possible paths. The grid doesn’t to be too dense or precise, it just must define good points in scene, and valid connections. Ideal distance between checkpoints is about 5 meters, although it may be also 1 or 20, depending on circumstances.

During game, each AI is capable to find connection between any two points in the mission, using specialized pathfinder class. This class is also used for traversing the found path, and return position where AI body should move to. It performs smoothing of the path, so that AI doesn’t seem to walk on the connections. Usually player even doesn’t notice there’s any checkpoint grid.

 

Sensory system

This system incorporates watching and hearing. Both are done using different techniques.

Watching is performed using list of possible seen objects (those enough close to AI), which is updated in some time intervals (e.g. once each three seconds). Each entry in the list is examined (again not in each tick), if it’s in field of view, and finally collision test is performed to see if it’s not obstructed.

Hearing is used by collecting sound information. Each sound played in the mission emits information, which is spread to actors interested in hearing (typically humans and animals). The hearing module is able to concatenate sound sources coming from similar direction, and performs some level of attenuation of sound’s importance over time.

 

Body movement

This is quite complex part of AI actor – it performs animation blending and updating of AI’s body. One possible approach is the simulation of the body using common class, which may be utilized by both humans and animals. The body has natural behavior, such as inertia, current fall speed, etc. The AI just issues commands to the body, for example desired movement direction, and body performs actual movement and animation over time. Also, appropriate animations are blended together in order to perform actual appearance of movement, along with blending of other animations performed on the body.

 

Dying

For dying, AI may use physics, which realistically animates body parts out of control of living organism.

 

Decision making

With all these inputs and outputs, there should be a module managing this all. The module should process inputs, perform decision, and issue commands to outputs. This decision module is a key to intelligent AI. It may also utilize scripts for personalized behavior or particular AI, or define general AI behavior in a global table, and perform decision based on “personality” of the AI character.

 

 

Sound playback

 

Sounds are ideally played in one of these ways:

 

Static sounds

The sounds put directly into the mission at design time. These sounds are at appropriate places, and either they are already playing (and looping), or they’re triggered to start playing at appropriate time.

 

Environment sounds

These are sounds created dynamically, for example footsteps of human characters, sound of storm, etc. Usually, someone creating such sound (for example human actor) doesn’t want to care further about the sound (anyway the sound resources must be kept in memory while the sound is playing), so ideally this task is performed by gamemission tick class. It may be implemented using a function, which creates requested sound, sets its parameters, and puts it into list of played sounds. Then, during Tick, gamemission checks the sounds and releases the ones which finished playing.

This way, it is comfortable for any actor or script to play a sound quickly.

 

Specific sounds

These may be for example voice of a human, engine sound of vehicle, etc. Usually belong are sounds, which are created once for particular actor, or other item. These sounds are created and managed entirely by the actor which uses them. When the actor is destroyed, so is the sound.
As an example, such sound may be human’s voice, used for playback of human’s speech, and other sounds originating from the human.

 

 

Mission development cycle

 

Here’s a proposed development cycle of a game mission:

 

Modeling

Of course, it starts with an idea. To turn the idea into reality, the mission should be sketched, and then modeled in a 3D modeling package. The artists model only unique, non-repeating geometry. They also texture-map the geometry, and export it into a mission file. As the mission is being created, basic light setup may be done in the game editor.

The artist may keep working on the mission modeling until desired quality and content is created.

Simultaneously, the artist may create models – microscenes, which are put into the scene in the editor one or multiple times. The models are set up either by artist, or designers.

 

Lighting

Lighting is done in the game editor. As work on mission progresses, lighting is updated and fine-tuned, and at some point, lightmapping is set up and computed. This also includes setting of other information, such as where dynamic shadows will be cast, etc.

 

Sounds

A work for sound designer. The sound design may be done in appropriate time, when the mission file is free, or simultaneously with other work, and later merging the results of sound design to the mission using the merge tool.

 

Collision setup

Another part is settings of collisions. If planned well, most models will have assigned collision objects directly in the modeling package, so that when they are reused, the contain collision information automatically. For terrain and static objects, this must be defined in the editor. This also involves assignment of logical materials to scene geometry. Again, for models this may be done in 3D modeling program.

 

AI setup and scripting

During this phase, designer places entities, AIs, active objects, etc into the scene. It assumes that most of other work has already been done.

Designer writes scripts, cinematic sequences, tests behavior, etc.

 

Note that this is just a proposal how development on a mission could advance. There’re many impact on actual workflow, so this idealistic proceeding doesn’t be always the best approach. Also, many times some resources, such as parts of program, sounds, or other parts, are not prepared at given time, so the development of a mission may also be significantly different than written here, anyway still be successful.
