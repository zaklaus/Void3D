Animations

 

How animations are used

 

The Insanity animation system uses properties of I3D_frame object. The animations are performed by changing position, rotation and scale of the frame. Since all 3D objects placed in a scene inherit from i3D_frame, it is possible to animate any 3D object. Additionally, for some objects it is possible to animate additional characteristics, such as light power, visibility (alpha), sound volume, etc.

 

The mentioned basic characteristics of a 3D object – position, rotation and scale, are inherited by a linked frames, so hierarchical animation system is implemented by the nature how Insanity uses scene hierarchy. The result of applying these properties is a transformation 3x4 matrix, which is used for transformation of visual’s vertices from local coordinates to world coordinates, and the matrix also affects transformation matrix of all linked frames.

 

The animation is a process of applying animated values during some time period, typically adjusting the properties in every update tick.

 

 

 

Types of animations

 

It is possible to animate 3D objects in several ways:

  • programmatically (for example moving characters using the player’s input)
  • importing and applying keyframe animations
  • using physics

 

Usually these methods are combined in a single game.

 

 

Animating objects programmatically

 

3D objects in a game may be animated using single algorithm, for example for opening a door, programmers may apply rotational value directly onto given 3D frame. This is accomplished using the method described above – applying animated position, rotation or other value onto frame over time. This method of animation is fully computed by application.

 

Similarly, if a physics engine is used to simulate physical behavior, it should apply the results of its computations directly onto animated frames.

 

 

Keyframe animations

 

In this section, we’ll focus mostly on keyframe animation system, which is built in the engine. The system is based on sample values (keys) placed on a time axis. The key value define some fixed values at specific time, while the animation system automatically computes the values between the keys. This system is well known from modeling packages, and Insanity is able to import and play back the keyframe animations.

 

The properties associated with each animation key are:

  • time (in milliseconds)
  • actual key value (position, rotation, or different)
  • easiness
  • smoothness

 

Key value represent the actual value, which will be applied when the time of animation is exactly at the key position. Additional values specify how data values between adjacent keys will be computed.

 

Easiness specify time morphing between two keys, and is represented by two values - easy_from and easy_to. For each segment between two keys, first key’s easy_from and next key’s easy_to values are used to compute time of acceleration from first key and de-acceleration towards the second key. The values are in range from 0 (no time morphing) to 1 (maximal time morphing).

 

Smoothness is defined by three values – tension, continuity and bias. This system is also known as TCB. The values specify curve, by which animation is computed between two keys. Bias value is kept only for compatibility reasons, and is not used by the engine. Other two values are used similar to how they are used in modeling packages.

 

Using the keyframe animations, it is possible to create and play animations, which look exactly same when they are animated in an animation editor, and when they are played back. This system is also suitable for importing of motion-capture animations.

 

The frequency of keys

In general, the number of keys, and time between them, does not matter much to the system. Having too many keys in animation (or too small value between keys, e.g. less than 10ms) may be uncomfortable for these reasons:

-          such animations are hard to edit, because you have to deal with too many keys

-          they consume more memory when loaded

-          they take more CPU time to process (although this impact is very low)

Considering all this, it is suggested to use animations with appropriate number of keys and their spacing. For example, a looped animation of human walking may consist of 16 keys, spaced about 60ms among the keys. When using motion-captured animations, it is good to filter out redundant keys prior to exporting the animation in the engine.

 

Insanity stores one or more animation tracks for single animated object in a I3D_keyframe_anim interface. This is a source information for an animation – animation tracks with key data.

 

 

 

Static poses

 

Static animation pose is used for definition of exact pose, which is not animated over time. Each pose may define position, rotation, scale or other value. Although pose for single object doesn’t make much sense, poses are utilized mainly with animation sets and animation blending.

 

 

 

Animation sets

 

When an animation is applied onto a complex object, such as character consisting of many 3D frames (joints, and others), the animation consists from animations for multiple objects, animated synchronously. For this reason, there’s a concept of set of animations, called animation set. The animation set contains one or more animation (either keyframe animation, or static pose). It also maintains additional information associated with each animation – a logical link, which is a string containing a name of object, to which the animation should be applied. You may read more about applying animations later.

 

The animation set is entity, which may be loaded from disk file (for example .i3d file), because animation files usually contain animation tracks for a group of 3D objects. When animation is loaded from a file, only animation tracks are read, completely ignoring any geometry or object info. The only information which is retrieved and maintained is the logical link – name of object to which particular animation belongs.

 

 

 

Animation blending

 

Here we’ll discuss another technique – animation blending. Although keyframe animations provide very good way of describing and playing animations, for real-time interactive applications (games), animations are changed frequently. In order to avoid popping when animation is changed, we use a concept of animation blending, where animations from multiple tracks are blended together, and the result is then applied onto the frame.

Animation blending is also used for generating of animations composed of several other animations.

 

The animation-blending engine works only for animations applied onto models. It is not working with plain interpolators (I3D_interpolator).

The blending system utilized multiple stages, in which animations are processed, blended together, and the result is applied onto 3D object.

Model’s frames are the frames owned by model. Animation set is one set of animations – I3D_animation_set. Operator defines operation made on stage, which defines how results of previous stages are blended with results of current stage. The stages are evaluated from left to right. Final results after evaluating all stages are applied onto animated frame.

 

The stage number is any number expressed dword value. The range of stage indices is not limited. The stage number is used only for identification and sorting of stages. If animation in particular stage is cleared or not set by the animation set, the stage is ignored.

 

Blending operation

This specifies actual operation performed on data. It is always associated with one stage, and defines how blending occurs with current results.

Following blend modes are defined:

 

-          SET - replace previous value by new value

-          ADD - add new value (scaled by scale factor) with previous value

-          BLEND - blend new value (scaled by scale factor) with previous value using blend factor

-          ADD - damp previous value by damp factor, and add current value (damp factor is actually a weight value, associated with animation and stored in animation set)

 

Additionally, the system allows simple precedence. Suppose you’d like to perform following blending operation:
result = (A + B) * (C * D)
where operation + specifies adding, and operation * specifies blending. You want first evaluate pairs of animations together, and then blend the results. To achieve this, you may specify logical braces when setting animation into particular stage. To learn more about it, see I3D_model::SetAnimation.

 

 

 

Applying animations

 

Animations in the engine are applied using interpolators. An interpolator is a class responsible for animating time value, traversing the time axis on an animation track, compute data values and apply the results onto 3D objects which it animates. Separate interpolator is used for animation of each frame.

There’re several ways how animations may be applied:

-          Loading a scene or model containing animation. This method is used when you want incorporate animation directly into scene or model – for example animated flag.

-          Creating interpolator, and apply animation to it. This is uncomfortable method of setting animations, not used very often. It is possible because of presence of interpolators in the system (which perform the animation if loaded scene contains some animated objects). Usually it is better for simple animations to compute animations programmatically and directly apply the values onto frame.

-          Loading animation set and applying it onto model. This is the most common use of animation system.

 

Further we’ll discuss the third mentioned way of applying animations. The I3D_model interface is designed to work directly with I3D_animation_set interface. The I3D_model contains a complex animations system, allowing blending on multiple stages.

To set an animation onto a model, you call I3D_model::SetAnimation method, with pointer to I3D_animation_set interface. The method sets the animations from the set onto objects which are owned by the model, and performs the linking of animation with model’s frame using the name link associated with each animation in the animation set. Only those animations from the set, whose names match a name of a frame in the model, are applied.

I3D_model also provides a range of methods for controlling how animations are played, and allows adjusting various parameters of animation stages. Using this system you may perform sophisticated animation blending. See documentation of I3D_model for details.

 

 

 

Updating animations

 

Somewhere in the pipeline, all played animations must be evaluated and applied onto animated objects. This is usually performed once during game update (tick). The updating is performed using the Tick method of animating.

If you created your own interpolator (I3D_interpolator), you’re responsible for calling its Tick method.

If you created a model and applied an animation on it, you’re responsible for calling its Tick method.

It is also necessary to call Tick method on the rendered scene, so that all animations containing in it are updated (scene keeps its own interpolators for performing animations on objects loaded from a scene file).

 

The Tick method of I3D_model or I3D_scene updates all animated interpolators running inside the class.

 

Basically it means you need to call Tick on all created objects which have Tick method. But there’s an exception. The I3D_scene interface is capable of updating all frames, which it owns – they are those which are added to scene’s container using the I3D_scene::AddFrame. If such added frame (typically model) contains animation, the scene takes care about it, and calls its Tick when you call the Tick on scene.

Similarly, the I3D_model is also a container, and if you add frame (typically model) to its container using the I3D_model::AddFrame method, the model takes care of the update when you call its Tick.

 

So or all models inserted into the scene in the editor, containing animations, the scene owns them, and thus updates their animation. You only need to take care of animated models which are created programmatically during game. Luckily, if there’s done wrong, it may be easily spot – objects on which Tick is not called will simple not animate, and objects onto which Tick is called multiple times will be animated faster.

 

 

 

Animation callbacks

 

The system supports animation callbacks, which are called during Tick-ing appropriate class. For example you may provide callback when calling Tick on a model, in order to process special messages.

The callbacks are used for these reasons:

 

Signaling the end of animation on particular stage.

This way you may detect if animation has finished, and perform appropriate action (e.g. play other animation) without the need to periodically check if animation in some stage finished after each Tick.

 

Sending message when note key is being processed.

One of animatable tracks is a note track, which is a track consisting of string values placed on a time axis. The note track is not applied onto animated frame in any way, and its only usage is by using callback, signaling that a note key has been encountered on the time axis during the Tick. Note tracks are good for signaling various events, that should happen in given time of animation. The meaning of the note string is fully user-defined – you may encode whatever meaning into the note text that you need.
